Reversort

mod01lec01 - Reversort
▶ Watch on YouTube ↗

Explanation

Reversort is a concrete sorting algorithm you're asked to analyze — not optimize, just understand deeply. The key skill here is reading an algorithm's pseudocode and reasoning about its invariants: what's true before each step, what changes, and how to measure some quantity (here, cost) as the algorithm runs. This kind of "simulate and count" thinking comes up constantly in competitive programming.

The mechanism is simple: the algorithm loops over positions $i = 1, 2, \ldots, n-1$. In each iteration it finds the minimum element in the suffix $A[i \ldots n]$, then reverses the subarray from $A[i]$ up to and including that minimum. The reversal length is $(j - i + 1)$ where $j$ is the minimum's current index, and that's the cost for this step. After the reversal, the minimum lands at position $i$ and is frozen — exactly like selection sort, but with a reversal instead of a swap.

Tiny example: array [3, 1, 2]. Step $i=1$: minimum of [3,1,2] is $1$ at index 2, reverse [3,1] → [1,3,2], cost $2$. Step $i=2$: minimum of [3,2] is $2$ at index 3, reverse [3,2] → [2,3], cost $2$. Total cost $= 4$. The sorted result is [1,2,3].

The deeper insight the problem builds toward is about the range of possible costs. At step $i$, the reversal length is at least $1$ (minimum is already in place) and at most $n - i + 1$ (minimum is at the very end). So total cost has a hard lower bound of $n - 1$ (one unit per step) and a hard upper bound of $\sum_{i=1}^{n-1}(n - i + 1) = 2 + 3 + \cdots + n = \frac{n(n+1)}{2} - 1$. Any cost strictly between those bounds can be achieved by "spending" exactly one extra unit in some steps — this is the constructive argument the problem asks you to formalize.

Common beginner pitfalls:

  • Off-by-one on the loop bound. The loop runs to $n-1$, not $n$ — the last element is already in place after the $n-1$st step. Iterating to $n$ either crashes or double-counts cost.
  • Confusing reversal length with reversal count. Cost is the length of each reversal, not the number of reversals. A length-1 reversal (element already in place) still costs 1, not 0.
  • Forgetting indices shift after each reversal. When you reverse in place, elements move. If you're simulating, always re-scan the array from position $i$ each step — don't cache the minimum's position from before the previous reversal.
  • Greedy construction going negative. When constructing a permutation for target cost $C$, you distribute the "extra" cost $C - (n-1)$ across steps greedily from left to right. If you overshoot and try to assign more extra than a step allows, you'll get a negative remainder — check the feasibility bounds first.

▸ Easier 1 cses Apartments ↗
Tags: greedy, sorting, two pointers
Why this problem

Drills Reversort's core inner-loop sub-skill in isolation: scan a sorted sequence and greedily match each element to its minimum qualifying partner — same left-to-right minimum-finding loop, without the reversal-cost bookkeeping.

▸ Style (canonical pattern + twist)
Canonical pattern: Two-pointer scan on two sorted arrays — a matching problem between two sequences with a tolerance window.
Twist: Textbook application; the only subtlety is identifying the correct pointer-advance rule at each step and arguing why a greedy left-to-right match is optimal.
Builds: The foundational greedy instinct: "assign the cheapest available resource that satisfies the current demand" — a pattern that recurs in interval scheduling, job assignment, and bipartite matching approximations.

Hint ladder (4 hints)

▸ Hint 1
Sort both lists on paper and try the example by hand. Notice: if applicant $a_i$ cannot be satisfied by the smallest remaining apartment, can any applicant with a smaller desired size be satisfied by it? Think carefully about what that implies for the order in which you should attempt matches.
▸ Hint 2
Once both arrays are sorted ascending, for applicant $a_i$ the acceptable apartment range is $[a_i - k,\ a_i + k]$. If the current smallest unexamined apartment $b_j$ falls below $a_i - k$, it is forever useless — no current or future applicant (all with larger desires) can ever be satisfied by it. What should happen to $b_j$?
▸ Hint 3
This is a greedy two-pointer: pointers $i$ over sorted applicants, $j$ over sorted apartments. Three cases each step: if $b_j < a_i - k$ advance $j$ (apartment too small for everyone); if $b_j \le a_i + k$ match and advance both (valid fit); otherwise $b_j > a_i + k$ and the apartment is too large for $a_i$ — advance only $i$ (no apartment further right will be smaller). Count matches.
▸ Hint 4
Full algorithm: sort $a$ and $b$ in $O((n+m)\log(n+m))$, then a single $O(n+m)$ two-pointer pass. Always use sys.stdin for large inputs. 
import sys
input = sys.stdin.readline
n, m, k = map(int, input().split())
a = sorted(map(int, input().split()))
b = sorted(map(int, input().split()))
i = j = ans = 0
while i < n and j < m:
    if b[j] < a[i] - k:
        j += 1
    elif b[j] <= a[i] + k:
        ans += 1; i += 1; j += 1
    else:
        i += 1
print(ans)
Edge cases: $k=0$ (exact match only) works naturally; $k=10^9$ makes nearly everything valid — no special handling needed either way.
▸ Easier 2 cses Concert Tickets ↗
Tags: greedy, sorting, binary search, ordered set
Why this problem

Upgrades the sorted-scan to an explicit 'find and consume the largest ticket ≤ budget' problem using a multiset — the same 'locate minimum in remaining set and remove it' operation Reversort performs at each step, now requiring O(log n) per query with a data structure.

▸ Style (canonical pattern + twist)
Canonical pattern: Dynamic ordered-multiset with floor (predecessor) queries — each query asks for the largest remaining element ≤ a threshold, then removes it.
Twist: Tickets are consumed one-by-one in arrival order, so the pool shrinks after every served customer; static binary search on a fixed array is not enough.
Builds: Recognising when a problem needs a mutable sorted structure (Python SortedList / C++ multiset) instead of a plain sorted array.

Hint ladder (4 hints)

▸ Hint 1
For each customer you want the best available ticket — the one priced as high as possible without exceeding their budget. Work through the example by hand with the tickets sorted: [3 5 5 7 8], customers 4 8 3. Which ticket "belongs" to each customer, and does the arrival order matter for correctness?
▸ Hint 2
With tickets sorted, each customer's query is: find the largest value in the current pool that is ≤ t. This is a classical floor (predecessor) query. The complication is that the pool shrinks — a sold ticket can never be reused — so you can't just binary-search a fixed array.
▸ Hint 3
You need a structure supporting two operations in $O(\log n)$: (1) floor query — largest element ≤ t, and (2) delete that element. Python's SortedList from sortedcontainers does both, giving $O((n+m)\log n)$ overall. Use bisect_right(t) - 1 to locate the floor index.
▸ Hint 4
Full sketch: from sortedcontainers import SortedList; sl = SortedList(tickets). For each budget t: i = sl.bisect_right(t) - 1; if i < 0 append -1, else append sl[i] and call sl.pop(i). Read input with sys.stdin and output with sys.stdout.write('\n'.join(map(str, ans))) — a plain print inside a loop TLEs at $n = 2 \times 10^5$.
▸ Similar 1 codeforces 1600 Replace by MEX ↗
Tags: greedy, implementation
Why this problem

Sort a permutation using a two-operation greedy strategy (insert MEX at index 0, or swap a[0] with a[MEX]) — pushes Reversort's 'greedy-choose-next-operation-to-advance-sort' intuition into a harder setting requiring case analysis on which operation progresses the algorithm at each step.

▸ Style (canonical pattern + twist)
Canonical pattern: Greedy simulation — use the MEX operation itself as a placement tool, targeting the "home" array [0, 1, 2, …, n−1].
Twist: The operation is self-referential: replacing with MEX changes the MEX, so you must argue that progress is monotone and bounded within 2n steps — not obvious until you find the right invariant.
Builds: Recognizing when a greedy "fix the cheapest thing now" strategy never backtracks, and counting operations via a pairing argument rather than brute-force bounds.

Hint ladder (11 hints)

▸ Hint 1
Simulate the operation by hand on a small array (say $n=4$). What final state do you keep landing on? Is there one particular sorted array that seems to be the natural destination of all these replacements?
▸ Hint 2
From hint 1: every value is between 0 and $n$, so the tightest non-decreasing target is $[0, 1, 2, \ldots, n{-}1]$. Notice that in this target, value $i$ lives at index $i$. So ask: when can a single operation send some value directly to its home index?
▸ Hint 3
If the current MEX is $m < n$, that means $m$ is absent from the array while $0, 1, \ldots, m{-}1$ are all present. In the target $[0, 1, \ldots, n{-}1]$, where does $m$ belong?
▸ Hint 4
Since $m$ is missing and belongs at index $m$, replacing $a[m]$ with MEX (which equals $m$) puts $m$ exactly in its correct slot. Now ask: once $a[m] = m$, can this position ever go wrong again in a future operation?
▸ Hint 5
MEX can only equal $m$ again if $m$ is absent from the array. But $a[m] = m$ keeps $m$ permanently present. Since we only ever overwrite $a[\text{MEX}]$ and MEX can never be $m$ again, position $m$ is frozen forever. Each "MEX $< n$" step fixes exactly one position and that fix is irreversible.
▸ Hint 6
So the MEX $< n$ case is clean: greedily fix position MEX each time. But what if MEX $= n$? MEX $= n$ means $0, 1, \ldots, n{-}1$ are all present. The array has $n$ elements and all $n$ values $\{0,\ldots,n{-}1\}$ appear — by pigeonhole it's a permutation of $\{0,\ldots,n{-}1\}$. What's still possibly wrong?
▸ Hint 7
A permutation can be unsorted (e.g., $[1, 0, 2]$). You can only place $n$ (the current MEX) somewhere. If you write $n$ over $a[i]$ (where $a[i]$ was some value $v$), then $v$ is now evicted from the array. What is the new MEX immediately after this?
▸ Hint 8
After evicting $v$ from position $i$, the value $v$ is now missing, so the new MEX $= v < n$. You're back in the easy case — and in the very next step you can place $v$ at index $v$, permanently fixing that position.
▸ Hint 9
So "MEX $= n$" steps always travel in pairs with a following "MEX $< n$" fix step. Count: at most $n$ positions to permanently fix (each needs at most 1 fix step), each possibly preceded by one "MEX $= n$" step → at most $2n$ total operations. When MEX $= n$, which position should you displace? Any wrong position works; the simplest is the first $i$ where $a[i] \neq i$.
▸ Hint 10
Full algorithm: loop up to $2n$ times. Each iteration: (1) if the array is already sorted, stop. (2) Compute MEX. (3) If MEX $< n$: write $a[\text{MEX}] = \text{MEX}$, record 1-indexed operation MEX $+ 1$. (4) If MEX $= n$: find the first $i$ with $a[i] \neq i$, write $a[i] = n$, record operation $i + 1$. The loop always terminates before $2n$ steps.
▸ Hint 11
Code sketch (Python):
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    a = list(map(int, input().split()))
    ops = []
    for _ in range(2 * n):
        if a == sorted(a): break
        present = set(a)
        mex = 0
        while mex in present: mex += 1
        if mex < n:
            ops.append(mex + 1)
            a[mex] = mex
        else:
            for i in range(n):
                if a[i] != i:
                    ops.append(i + 1)
                    a[i] = n
                    break
    print(len(ops))
    print(*ops)

t = int(input())
for _ in range(t): solve()
Edge cases: if ops is empty, print(*ops) prints a blank line — that's valid. a == sorted(a) is $O(n \log n)$ inside a $2n$ loop but fine for $n \leq 1000$. For tight I/O use sys.stdout.write and '\n'.join(...) when scaling.
▸ Similar 2 cses Sliding Window Median ↗
Tags: sorting, data structures, sliding window, ordered set, heap
Why this problem

Instead of sorting once, maintain sorted order dynamically as a window slides through the input. Shares Reversort's 'track relative order across successive steps' skill but adds the twist that elements continuously enter and leave, requiring an ordered data structure rather than a single forward pass.

▸ Style (canonical pattern + twist)
Canonical pattern: Sliding window over a dynamically maintained sorted multiset — an order-statistic tree query (rank access) repeated $n$ times.
Twist: You need insert, delete-by-value, and rank-based access all in $O(\log k)$ simultaneously — a plain heap only exposes the extreme element and cannot reach arbitrary ranks.
Builds: Recognising when a problem needs richer ordered-structure capabilities than a heap provides, and reaching for Python's SortedList (or two-heap lazy deletion) as the right tool.

Hint ladder (13 hints)

▸ Hint 1
As the window slides one position to the right, three things must happen to the collection of $k$ elements. Think carefully about what those three operations are — and before reaching for any data structure, ask: which of the three is a plain sorted array or a heap fundamentally bad at?
▸ Hint 2
The three operations are: (1) insert the new right element, (2) delete the element that fell off the left, and (3) read the element at a specific rank in sorted order. All three happen $n - k + 1$ times. Operations 1 and 3 are easy on different structures — the hard constraint is that all three must be fast simultaneously.
▸ Hint 3
A sorted array makes rank-access trivial (just index in), but deletion by value requires shifting — $O(k)$ per step, so $O(nk)$ total. With $n, k \le 2 \times 10^5$ that is $4 \times 10^{10}$ — far too slow. So the real bottleneck is: can we delete a specific value from a sorted collection in better than $O(k)$?
▸ Hint 4
A min-heap or max-heap can insert and delete-min in $O(\log k)$, but it can only expose the extreme element — it has no way to answer 'what is the element at rank $\lfloor(k-1)/2\rfloor$?' What structure supports insert, delete-by-value, and rank-based index access, all in $O(\log k)$?
▸ Hint 5
That structure is a balanced BST augmented with subtree sizes — also called an order-statistic tree. In Python, SortedList from the sortedcontainers library is exactly this: sl.add(x), sl.remove(x), and sl[i] (rank lookup) are all $O(\log k)$. This is the right tool.
▸ Hint 6
From hint 5 we have the structure — now wire up the window. For each index $i$, call sl.add(a[i]). Once $i \ge k$, also call sl.remove(a[i - k]) to evict the element that left the window. After both operations, sl always holds exactly the current $k$ elements in sorted order.
▸ Hint 7
Which index into the sorted list is the median? For $k$ elements (0-indexed 0 to $k-1$), the problem says for even $k$ take the smaller of the two middles. That is always index $\lfloor(k-1)/2\rfloor$. Check: $k=3 \Rightarrow$ index 1 (true middle); $k=4 \Rightarrow$ index 1 (smaller of the two middles). Verify with the first example window $[2,4,3] \to$ sorted $[2,3,4]$: index 1 = 3. ✓
▸ Hint 8
What about duplicate values? SortedList stores every insertion separately — it is a sorted multiset, not a set. So sl.remove(x) removes exactly one copy of $x$, which is correct behaviour. Duplicates require no special handling; the invariant is maintained automatically.
▸ Hint 9
Build the window incrementally rather than pre-filling: start with an empty SortedList, add one element per iteration, and output only once len(sl) == k (equivalently, once $i \ge k-1$). This sidesteps off-by-one errors and handles the edge cases $k=1$ and $k=n$ uniformly.
▸ Hint 10
sortedcontainers is available on CSES and most competitive programming judges but is not in the Python standard library — you must write from sortedcontainers import SortedList explicitly. If a judge lacks it, the fallback is two heaps with lazy deletion (described in the next hint).
▸ Hint 11
Two-heap lazy deletion (worth knowing for restricted judges): keep a max-heap lo for the lower half and a min-heap hi for the upper half. Mark departing elements in a deletion counter; clean them lazily when they surface at a heap top. Rebalance sizes to $\lfloor k/2 \rfloor$ and $\lceil k/2 \rceil$ after each insert/evict. Same $O(n \log k)$ complexity but roughly 3× more code — SortedList is the cleaner path when available.
▸ Hint 12
Time complexity: $O(n \log k)$ — each of the $n$ iterations does one add, one remove, and one index lookup, each costing $O(\log k)$. Space: $O(k)$ for the window. This comfortably fits within the $2 \times 10^5$ constraint.
▸ Hint 13
Code sketch (Python, SortedList path):
import sys
from sortedcontainers import SortedList

def main():
    data = sys.stdin.read().split()
    n, k = int(data[0]), int(data[1])
    a = list(map(int, data[2:]))
    sl = SortedList()
    med = (k - 1) // 2
    out = []
    for i in range(n):
        sl.add(a[i])
        if i >= k:
            sl.remove(a[i - k])
        if i >= k - 1:
            out.append(str(sl[med]))
    print(' '.join(out))

main()
Use sys.stdin.read() for fast input. Accumulate results in a list and ' '.join at the end — repeated print calls inside a loop are slow. Edge case $k=1$: sl[0] every step = the element itself. ✓
