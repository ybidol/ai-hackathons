Explanation

Most binary search problems you've seen so far look like "find x in a sorted array." But there's a more powerful pattern hiding in the same idea: binary search on the answer. Instead of searching through given data, you search through candidate answers and ask, for each candidate, "is this one feasible?" The trick that makes it work is monotonicity: if some answer $k$ works, then every answer larger than $k$ also works (or vice versa). Whenever you can spot that "yes/no" pattern flipping at exactly one threshold, binary search finds the threshold in $O(\log N)$ steps.

The mechanism is a clean three-part recipe. First, identify what you're searching over — usually "minimum number of days," "maximum capacity," "smallest distance," etc. Second, write a check(x) function that returns true if answer $x$ is achievable. Third, binary-search the range $[lo, hi]$ where $lo$ is obviously too small and $hi$ is obviously enough. The classic loop is: while $lo < hi$, take $mid = (lo + hi) / 2$; if check(mid), set $hi = mid$; else $lo = mid + 1$. At the end $lo = hi$ is the answer.

Why is this a leap forward? check(x) is often much easier to write than "compute the optimal answer directly." You're allowed to assume you have $x$ resources, $x$ days, $x$ budget — and just simulate or use a greedy. The hard optimization problem becomes a sequence of easier feasibility problems.

Tiny example: "What's the minimum integer $d$ such that a robot stepping $+2$ per second covers at least $17$ units?" Brute-force loop works, but binary-search on $d \in [0, 100]$ asks check(d) = (2 \cdot d \ge 17). check(8) is false ($16 < 17$), check(9) is true ($18 \ge 17$) — so answer is $9$. Trivial here, but the same skeleton handles problems where simulating a single $d$ takes $O(N)$ work and the answer can be up to $10^{18}$.

For Magic Ship specifically, the relevant insight is: more days never hurt you. If you can reach the island in $k$ days (the wind pushes you somewhere, and your remaining "free moves" cover the gap), you can also reach it in $k+1$ days — just use one extra free move and step back. That monotonicity is exactly what unlocks binary search on the day count.

Common beginner pitfalls to watch out for:

  • Off-by-one in the loop. Mixing up lo = mid vs lo = mid + 1 causes infinite loops. Memorize one shape and stick to it: for "smallest $x$ with check true," use hi = mid / lo = mid + 1.
  • Wrong upper bound. If $hi$ is too small, you'll return a false negative. Pick $hi$ generously — for Magic Ship, think about the worst case where wind cancels out and you must walk the whole Manhattan distance yourself.
  • Overflow. Day counts and coordinates here go up to $10^9$; their products and sums need long long (C++) or Python's arbitrary precision. (lo + hi) / 2 can overflow — prefer lo + (hi - lo) / 2.
  • Non-monotonic predicate. Before coding, prove to yourself that "if $k$ works, $k+1$ works." If the predicate flips back and forth, binary search is the wrong tool.
  • Expensive check. Total cost is $O(\log(\text{range}) \cdot \text{check cost})$ — keep check at $O(N)$ or better, or the overall solution will TLE.

▸ Easier 1 cses Factory Machines ↗
Tags: binary search
Why this problem

Isolates the 'binary search on answer + O(n) feasibility check' skeleton used in Magic Ship — binary search on time T, verify sum(floor(T/k_i)) >= n — with no 2D geometry, so you internalize the predicate structure before adding displacement.

▸ Style (canonical pattern + twist)
Canonical pattern: Binary search on the answer — search directly over the time value rather than over configurations.
Twist: Textbook application; the key step is recognising that "given a fixed time, can we hit the target?" is easy to answer, and that this feasibility is monotone in time.
Builds: The core pattern-recognition habit of asking "is there a threshold I can binary-search over?" — the most reusable idea in competitive programming.

Hint ladder (5 hints)

▸ Hint 1
If someone handed you a candidate answer $T$ seconds, how would you verify it without trying every possible schedule? Try computing something concrete for $T = 8$ in the example (machines with speeds 3, 2, 5).
▸ Hint 2
From hint 1: in $T$ seconds, machine $i$ independently completes $\lfloor T / k_i \rfloor$ products (it just runs its cycle as many times as it fits). The total you can produce is $\sum_i \lfloor T / k_i \rfloor$. Now ask: as $T$ grows, does this total ever decrease?
▸ Hint 3
From hint 2, the total is non-decreasing — adding one more second never reduces output. This means there is a clean threshold: every $T$ below some value is infeasible, every $T$ at or above it is feasible. What search strategy is designed exactly for this shape?
▸ Hint 4
Binary search on the answer. Set lo = 1, hi = min(k) * t (upper bound: the single fastest machine makes all $t$ products alone). At each mid, check sum(mid // ki for ki in k) >= t; if yes, shrink right (hi = mid); if no, shrink left (lo = mid + 1). The answer is lo when the loop exits.
▸ Hint 5
Python code sketch — $O(n \log(\min(k) \cdot t))$:
import sys
input = sys.stdin.readline
n, t = map(int, input().split())
k = list(map(int, input().split()))
lo, hi = 1, min(k) * t
while lo < hi:
    mid = (lo + hi) // 2
    total = 0
    for ki in k:
        total += mid // ki
        if total >= t:   # early exit — no need to keep summing
            break
    if total >= t:
        hi = mid
    else:
        lo = mid + 1
print(lo)
Edge cases: Python big-ints handle the large product min(k)*t (up to ~$10^{18}$) without overflow. The early-exit once total >= t is important for large $n$ — without it you may TLE on worst-case inputs.
▸ Easier 2 codeforces 1500 Maximum Median ↗
Tags: binary search, greedy, prefix sums
Why this problem

Binary search on a target value with a prefix-sum scan to count required operations — the same search-then-check pattern as Magic Ship applied to a 1D array, building predicate intuition before the 2D wind-drift version.

▸ Style (canonical pattern + twist)
Canonical pattern: Binary search on the answer — asking "is median ≥ m achievable?" instead of "what is the best median?", then checking feasibility with a greedy prefix sum.
Twist: The median's rank position in the sorted array makes direct greedy murky, but the monotone cost function (higher target = more operations) makes binary search feel inevitable once you reframe.
Builds: The core instinct of "binary search on the answer" — recognizing when a maximization problem has a monotone yes/no structure and translating it into a cost feasibility check.

Hint ladder (5 hints)

▸ Hint 1
Instead of asking "what's the maximum median I can reach?", try flipping it: ask "can I make the median at least $m$?" As $m$ increases, does this get easier or harder? What does that monotonicity imply about how to search for the answer?
▸ Hint 2
After sorting the array, the median sits at index $n/2$. For the median to be $\geq m$, you need the element at that index (and everything to its right) to each be $\geq m$. Elements to the left of the median don't matter — even if you boost them, they don't help the median. Which elements is it cheapest to push up to $m$?
▸ Hint 3
The minimum number of operations to achieve median $\geq m$ is $\sum_{i=n/2}^{n-1} \max(0,\; m - a[i])$ after sorting. If this sum $\leq k$, target $m$ is feasible. Notice this cost is non-decreasing in $m$ — confirming the monotone structure from hint 1.
▸ Hint 4
Binary search on $m$ over the range $[a[n/2],\; a[n/2] + k]$. Use the upper bound $a[n/2] + k$ because even if you pour all $k$ operations into the median element alone, you can't exceed that value. Each feasibility check runs in $O(n)$, giving $O(n \log k)$ total.
▸ Hint 5
Python sketch — watch for the "find maximum" binary search template (add 1 to mid, move lo up):
import sys
input = sys.stdin.readline

def solve():
    n, k = map(int, input().split())
    a = list(map(int, input().split()))
    a.sort()
    mid = n // 2

    def feasible(m):
        cost = 0
        for i in range(mid, n):
            if a[i] < m:
                cost += m - a[i]
                if cost > k:  # early exit
                    return False
        return True

    lo, hi = a[mid], a[mid] + k
    while lo < hi:
        m = (lo + hi + 1) // 2  # +1 avoids infinite loop when lo+1==hi
        if feasible(m):
            lo = m
        else:
            hi = m - 1
    print(lo)

solve()
Edge cases: $n=1$ trivially returns $a[0]+k$; the early-exit in feasible keeps each check fast in practice.
▸ Similar 1 codeforces 1800 The Best Vacation ↗
Tags: binary search, prefix sums, two pointers
Why this problem

Sliding window on a doubled circular array with prefix sums — directly exercises the cyclic prefix-sum sub-skill Magic Ship uses to compute cumulative wind displacement after d full wind cycles.

▸ Style (canonical pattern + twist)
Canonical pattern: Sliding window / two pointers over a doubled circular array with prefix sums.
Twist: Each month contributes variable hugs (day $j$ gives $j$ hugs), so partial-month boundaries require a closed-form triangle formula; the cyclic calendar requires array doubling to linearize year wrap-arounds.
Builds: Recognizing when the optimal window always aligns to a natural boundary (month endpoints here), reducing a continuous positional search to a discrete enumeration — the core insight behind many sliding-window problems.

Hint ladder (10 hints)

▸ Hint 1
Work out the first example by hand: days are {1,1,2,3,1} and you need 2 consecutive days. Try every starting position and list the hugs. Notice that whenever a month is fully inside your window, there is no choice — you always take every day of it. Where is the only flexibility?
▸ Hint 2
The only flexible piece is the starting partial month. Since day $j$ gives $j$ hugs, and you can only take $k$ days from a month, which $k$ days maximize hugs — the first $k$ or the last $k$? Does this tell you something about the shape of the optimal left boundary?
▸ Hint 3
So the optimal window always looks like: [last $k$ days of some month $L$] + [complete months $L{+}1, L{+}2, \ldots, R$], ending exactly at the end of month $R$. This means you only need to enumerate right endpoints — one candidate per month — and find the best matching left end for each.
▸ Hint 4
The trip can span a year boundary. The cleanest fix: duplicate the months array to length $2n$. Any window that would wrap around now appears as a contiguous window in the doubled array, and you only need to consider windows in the first $n$ "slots" for the right endpoint (or all $2n$ — it's safe either way).
▸ Hint 5
Define prefix sums over the doubled array: $P[i] = d_0 + d_1 + \cdots + d_{i-1}$ (cumulative days) and $H[i] = \sum_{j < i} \frac{d_j(d_j+1)}{2}$ (cumulative hugs for complete months). For complete months $[l \ldots r]$, the day count is $P[r+1] - P[l]$ and the hug count is $H[r+1] - H[l]$.
▸ Hint 6
For a fixed right endpoint $r$, you want the largest set of complete months $[l \ldots r]$ that still fits in $x$ days (i.e., $P[r+1] - P[l] \le x$), then fill the leftover $\text{rem} = x - (P[r+1] - P[l])$ days from the tail of month $l{-}1$. Can you write a closed form for the hugs from the last $\text{rem}$ days of a $d$-day month?
▸ Hint 7
The sum of the last $k$ days of a $d$-day month is $d + (d-1) + \cdots + (d-k+1) = \frac{k(2d - k + 1)}{2}$. You now have all pieces to evaluate any candidate window. The total hugs for right endpoint $r$ with left boundary starting at $l$ is $(H[r+1] - H[l]) + \frac{\text{rem}(2d_{l-1} - \text{rem} + 1)}{2}$.
▸ Hint 8
Now think about efficiency. As $r$ increases by one (moving right), can $l$ ever move left? Adding a new month on the right only makes the window larger, so $l$ can only stay or advance right. This monotonicity means both pointers together traverse the array at most once — $O(n)$ total work.
▸ Hint 9
Two-pointer algorithm: initialize $l = 0$; for each $r$ from $0$ to $2n-1$, advance $l$ forward while $P[r+1] - P[l] > x$. After the loop, months $[l \ldots r]$ all fit in $x$ days; compute $\text{rem}$ and the partial-month hugs from month $l{-}1$; update the running maximum. Skip the partial term when $l = 0$ (no month to the left) or $\text{rem} = 0$.
▸ Hint 10
Python code sketch — edge cases and perf notes inline:
import sys
input = sys.stdin.readline

def solve():
    n, x = map(int, input().split())
    d = list(map(int, input().split()))
    d = d * 2          # linearize circular calendar
    m = 2 * n
    pdays = [0] * (m + 1)
    phugs = [0] * (m + 1)
    for i in range(m):
        pdays[i+1] = pdays[i] + d[i]
        phugs[i+1] = phugs[i] + d[i] * (d[i] + 1) // 2
    ans = 0
    l = 0
    for r in range(m):
        while pdays[r+1] - pdays[l] > x:
            l += 1
        rem = x - (pdays[r+1] - pdays[l])
        hugs = phugs[r+1] - phugs[l]
        if rem > 0 and l > 0:
            dl = d[l - 1]
            hugs += rem * (2 * dl - rem + 1) // 2
        ans = max(ans, hugs)
    print(ans)

solve()
Answers can reach ~$10^{16}$ (Python big-ints handle this fine). When $l=0$ and $\text{rem}>0$ the window isn't full yet — the maximum will be found once $r$ is large enough that a full $x$-day window exists.
▸ Similar 2 codeforces 1800 RPG Protagonist ↗
Tags: binary search, greedy, prefix sums, sorting
Why this problem

Enumerate one warrior's heavy-item count, binary-search the other's remaining capacity via sorted prefix sums — nested binary search with prefix-sum feasibility lookup, the natural harder extension of Magic Ship's single binary search + displacement check.

▸ Style (canonical pattern + twist)
Canonical pattern: Greedy prefix selection (always take lightest items) combined with binary search — enumerate one dimension, binary-search the other.
Twist: The hard constraint is that one person carries only swords and the other carries only shields — which collapses the combinatorial allocation into two independent prefix-count decisions.
Builds: The habit of "sort + prefix sums + binary search" as a unit; recognizing when an enumeration-over-one-variable trick reduces a 2D optimization to 1D.

Hint ladder (10 hints)

▸ Hint 1
Work through a tiny example by hand: 3 swords [2, 5, 9], 3 shields [1, 4, 7], person A can carry weight 10, person B can carry weight 8. You're trying to maximize total items carried. Notice that once you decide how many swords person A takes, is there any remaining choice in which swords they take?
▸ Hint 2
From hint 1: if you're taking $k$ swords to maximize count, you always want the $k$ lightest swords — heavier ones only hurt. So sorting swords ascending and reading a prefix is always optimal for a given count. The same logic applies to shields independently. Both people are just choosing a prefix of their sorted array.
▸ Hint 3
Here is the structural constraint that dominates everything: one person carries only swords, the other carries only shields. There is no mixing. So the entire allocation is determined by exactly two integers: how many swords one person takes, and how many shields the other takes — both from sorted prefixes.
▸ Hint 4
Build the prefix sums: $\text{ps}[k]$ = weight of the $k$ lightest swords, $\text{pp}[j]$ = weight of the $j$ lightest shields. Checking "can person A carry $k$ swords?" is now $O(1)$: just verify $\text{ps}[k] \leq W_A$. This means you can enumerate all valid $k$ in a single pass.
▸ Hint 5
From hint 4: for each valid $k$ (swords person A takes, with $\text{ps}[k] \leq W_A$), person B independently wants the largest $j$ such that $\text{pp}[j] \leq W_B$. Since $\text{pp}$ is strictly increasing (all weights are positive), this largest valid $j$ is found by binary search on the prefix-sum array.
▸ Hint 6
Now the shape of the solution is clear: sort both arrays ascending, build prefix sums. Enumerate $k$ from 0 to the largest valid sword count. For each $k$, binary-search $\text{pp}$ for the largest $j$ with $\text{pp}[j] \leq W_B$. Track $\max(k + j)$. Each loop iteration is $O(\log m)$, total loop is $O(n \log m)$.
▸ Hint 7
Critical: you also need to try swapping roles — person A carries shields (up to $W_A$), person B carries swords (up to $W_B$). Run the same enumeration with the arrays and weight limits exchanged. The final answer is the maximum across both role assignments. Missing this swap is the most common wrong answer.
▸ Hint 8
Verify your indexing convention: use a prefix-sum array of length $n+1$ where index 0 means "0 items, weight 0". Then $\text{bisect\_right}(\text{pp}, W_B) - 1$ gives the largest $j$ with $\text{pp}[j] \leq W_B$, and since $\text{pp}[0] = 0$, the result is always $\geq 0$ even when $W_B = 0$. This handles the edge case of one person carrying nothing automatically.
▸ Hint 9
Complexity: $O(n \log n + m \log m)$ for sorting. Two enumeration passes of length $n$ and $m$ respectively, each with an $O(\log m)$ or $O(\log n)$ binary search. Total $O((n+m)\log(n+m))$. With $n, m \leq 2 \times 10^5$, this is comfortably within limits.
▸ Hint 10
Final hint — Python code sketch with edge-case handling and performance notes:
import sys
from bisect import bisect_right
input = sys.stdin.readline

def solve():
    n, m, W1, W2 = map(int, input().split())
    s = sorted(map(int, input().split()))
    p = sorted(map(int, input().split()))

    ps = [0] * (n + 1)  # prefix sums of swords
    for i in range(n): ps[i+1] = ps[i] + s[i]
    pp = [0] * (m + 1)  # prefix sums of shields
    for i in range(m): pp[i+1] = pp[i] + p[i]

    ans = 0
    # Role 1: person 1 takes swords (W1), person 2 takes shields (W2)
    for k in range(n + 1):
        if ps[k] > W1: break
        j = bisect_right(pp, W2) - 1
        ans = max(ans, k + j)
    # Role 2: person 1 takes shields (W1), person 2 takes swords (W2)
    for k in range(m + 1):
        if pp[k] > W1: break
        j = bisect_right(ps, W2) - 1
        ans = max(ans, k + j)

    print(ans)

solve()
Perf note: sys.stdin.readline is essential here. The break in each loop is valid because prefix sums are monotonically increasing — once $\text{ps}[k] > W_1$ no larger $k$ can be valid.
