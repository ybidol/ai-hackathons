Simple Skewness

mod02lec08 - Simple Skewness
▶ Watch on YouTube ↗

Explanation

Many "find the best subset / pair / triple" problems are too big to brute-force directly, but they crack open once you fix one parameter and ask the easier question of how to optimize the rest. Here that pivot is the median: instead of searching over all $2^n$ subsets, you say "suppose the median is $a_i$ — given that, what is the best mean I can achieve?" Once that question is fast, you just try every candidate $a_i$.

The other half of the trick is sorting first. Sorted arrays turn vague conditions like "values below the median" or "the $k$ biggest" into clean range queries — they are simply prefixes and suffixes. So the recipe is: sort, fix a position $i$ in the sorted array, and use prefix sums (or a heap, or precomputed max-sums) to evaluate the best completion in $O(1)$ or $O(\log n)$.

For Simple Skewness in particular, an odd-size subset of size $2k+1$ with median $m$ must contain exactly $k$ elements $\le m$ and $k$ elements $\ge m$. To make the mean as large as possible without disturbing the median, you greedily pick the $k$ largest values from each side — closest to $m$ from below, farthest from $m$ above. Sorting and a couple of running sums make this evaluation cheap, and you sweep $k$ for every choice of $m$.

Tiny illustration of the pattern on a friendlier task — "pick 3 numbers with the largest sum whose median is fixed". On sorted $a = [1,2,5,7,9]$, if you fix the median as $5$, the best triple is $\{2, 5, 9\}$: take the largest element below $5$ and the largest above. The point isn't this trivial answer; it's that once $5$ is fixed, the remaining choice is forced and instant. Trying each possible median in turn gives the global optimum.

Common beginner pitfalls:

  • Forgetting to sort, or sorting and then using original indices — keep your indexing in one space.
  • Off-by-one on the median: for size $2k+1$ the median is at sorted position $k$ (0-indexed), and there must be exactly $k$ elements on each side, not $k$ vs $k+1$.
  • Double-counting the pivot when summing "below + median + above" — the median itself is one element, not part of either side.
  • Re-doing $O(n)$ work inside the loop. The whole speedup comes from precomputing prefix sums or "best $k$ on each side" tables once, then reusing them per candidate.
  • Missing structure in $k$. For a fixed median, the score as a function of subset half-size $k$ is often unimodal — it rises, peaks, then falls. Recognizing this lets you ternary-search $k$ instead of trying every value, which can be the difference between TLE and AC.

Tags: sorting, median
Why this problem

Directly builds the intuition that the median minimizes total absolute deviation—the exact property 600/D exploits when it fixes one element as the median and maximizes the surplus sum of the remaining window.

▸ Style (canonical pattern + twist)
Canonical pattern: Minimize the sum of absolute deviations — choose a target value $t$ and pay $\sum |p_i - t|$.
Twist: Recognizing that the optimal $t$ is the median, not the mean, is the entire problem. The math is textbook once you see it.
Builds: L1 vs L2 optimization intuition — why median beats mean for absolute cost, and why sorting unlocks the answer.

Hint ladder (5 hints)

▸ Hint 1
Try [1, 2, 10] by hand. The mean is ~4.3 — does using 4 as the target actually feel cheapest? Try target = 1, 2, 10 and compare. What do you notice about where the minimum lands relative to the sorted list?
▸ Hint 2
The total cost is $\sum |p_i - t|$. Imagine sliding $t$ continuously from left to right along the number line. Each unit step right increases cost by +1 for every stick to your left and decreases it by -1 for every stick to your right. What condition tells you the cost has stopped decreasing?
▸ Hint 3
Cost stops falling exactly when the number of sticks to your left equals the number to your right — i.e., at the median. This is the classical result: the median minimizes the sum of absolute deviations (contrast with the mean, which minimizes the sum of squared deviations). Any movement away from the median moves more sticks 'against' you than 'with' you.
▸ Hint 4
Algorithm: sort the array, set $t = p[n // 2]$, return $\sum |p_i - t|$. For even $n$ any value between the two middle elements is optimal — but since stick lengths are integers, either middle element gives the same total cost. Complexity: $O(n \log n)$ for sorting, $O(n)$ for the sum.
▸ Hint 5
Python code sketch — watch the output size: the answer can reach $2 \times 10^{14}$, but Python integers handle that natively.
import sys\ninput = sys.stdin.readline\n\nn = int(input())\np = sorted(map(int, input().split()))\nmedian = p[n // 2]\nprint(sum(abs(x - median) for x in p))
Edge cases: $n=1$ → cost 0 (already done). All equal → median equals every element, sum is 0. No special-casing needed.
▸ Easier 2 codeforces 1300 Electrification ↗
Tags: sorting, two pointers, sliding window
Why this problem

Practices sort + two-pointer range scan to count elements inside a sliding window—the same primitive 600/D uses when, for a fixed median element, it determines how many original-array positions fall on each side of a candidate window.

▸ Style (canonical pattern + twist)
Canonical pattern: Fixed-size sliding window on a sorted array.
Twist: The input is already sorted, and you need to recognize that the $k+1$ nearest points to any $x$ always form a consecutive block — that reframes the whole problem.
Builds: The habit of converting "find optimal point" questions into "find optimal window" questions on sorted sequences.

Hint ladder (5 hints)

▸ Hint 1
The array is already sorted. Try a small example: $n=4$, $k=1$, points $[1, 3, 8, 10]$. Pick a few candidate values of $x$ and compute which two points are closest to each one. What do you notice about where in the sorted array those two closest points sit?
▸ Hint 2
From hint 1 you may have noticed: for any $x$, the $k+1$ nearest points always form a consecutive block in the sorted array. Convince yourself why — if you skip a point, there's always a closer one in between.
▸ Hint 3
From hint 2, the question becomes: for each window of $k+1$ consecutive points $[a_i, a_{i+k}]$, what is the best possible $f_k$ value achievable by placing $x$ optimally inside that window? Express it in terms of $a_i$ and $a_{i+k}$ only.
▸ Hint 4
The best $x$ for window $[a_i, a_{i+k}]$ is the midpoint, giving $f_k = \lceil (a_{i+k} - a_i) / 2 \rceil$. Since this grows with the window span, just slide a fixed-size window of $k+1$ elements, track the minimum span $a_{i+k} - a_i$, and output $(a_i + a_{i+k}) // 2$ for the best window. $O(n)$ per query.
▸ Hint 5
Python sketch — use sys.stdin and '\n'.join for the large $T$:
import sys
input = sys.stdin.readline
T = int(input())
out = []
for _ in range(T):
    n, k = map(int, input().split())
    a = list(map(int, input().split()))
    best_x = (a[0] + a[k]) // 2
    min_span = a[k] - a[0]
    for i in range(1, n - k):
        span = a[i + k] - a[i]
        if span < min_span:
            min_span = span
            best_x = (a[i] + a[i + k]) // 2
    out.append(str(best_x))
print('\n'.join(out))
Edge cases: $k=0$ means any $a_i$ works (span 0, distance 0). The problem says multiple answers are fine, so the first valid midpoint is acceptable.
▸ Similar 1 codeforces 2000 Devu and his Brother ↗
Tags: binary search, greedy, sorting, ternary search
Why this problem

Same 'sort + ternary-search over a threshold' skeleton as 600/D: sort both arrays, ternary-search on the split value t minimizing #{a[i]>t}+#{b[j]<t}—unimodal objective, identical outer structure, without the nested median constraint.

▸ Style (canonical pattern + twist)
Canonical pattern: ternary search on a convex cost function over a continuous threshold variable.
Twist: the threshold x is not given — you must invent it as an auxiliary variable, then recognize that the total cost is a convex (bowl-shaped) function of x. That convexity is the non-obvious unlock.
Builds: spotting convexity in sum-of-costs problems; combining ternary search with prefix-sum + binary-search evaluation for an $O(n \log n + \log V \cdot \log n)$ solution.

Hint ladder (10 hints)

▸ Hint 1
The condition says min(a) ≥ max(b). Picture the 'finished' state: there exists some level x such that every element of a is at or above x and every element of b is at or below x. You get to choose x freely. Try a few concrete values of x on a small example — does the total cost vary a lot between choices, and does it feel like there is a 'sweet spot'?
▸ Hint 2
From hint 1: for any fixed threshold x, the cost is fully determined. Every a[i] below x must be raised to x (paying x − a[i] per element); every b[j] above x must be lowered to x (paying b[j] − x per element). Write this as $f(x) = \sum_i \max(0,\, x - a_i) + \sum_j \max(0,\, b_j - x)$. This formula works for any integer x, not just values already in the arrays.
▸ Hint 3
Focus on just the first sum: $C_a(x) = \sum_i \max(0, x - a_i)$. What happens to $C_a$ when x increases by 1? Every a[i] that was already below x now costs 1 more operation, and any a[i] equal to x is newly 'below the bar'. So $C_a(x)$ is non-decreasing in x — committing to a higher threshold for a never saves you operations on the a side.
▸ Hint 4
By the same logic, $C_b(x) = \sum_j \max(0, b_j - x)$ is non-increasing in x: raising x can only shrink or eliminate gaps between b elements and the bar. You now have $f(x) = C_a(x) + C_b(x)$ — a non-decreasing term plus a non-increasing term. What shape must their sum have as a function of x?
▸ Hint 5
$f(x)$ is the sum of two convex piecewise-linear functions, so $f$ itself is convex: it has a single valley (possibly flat) and no other local minima. A convex function over an integer range can be minimized with ternary search in $O(\log V)$ evaluations (V ≈ $10^9$). The remaining question: can you evaluate $f(x)$ for a single x faster than $O(n + m)$?
▸ Hint 6
To make each evaluation fast, sort a ascending and precompute prefix sums pa[], where pa[k] = a[0] + ... + a[k−1]. For a given x, the elements of a that are strictly below x form a contiguous prefix of the sorted array. Find the prefix length i = bisect_left(a, x) in $O(\log n)$, then $C_a(x) = i \cdot x - \text{pa}[i]$.
▸ Hint 7
Do the same for b: sort b ascending and build prefix sums pb[]. Elements of b strictly greater than x form a suffix — find where the suffix starts with j = bisect_right(b, x), also $O(\log n)$. Then $C_b(x) = (\text{pb}[m] - \text{pb}[j]) - (m - j) \cdot x$. Each evaluation of $f(x)$ now costs $O(\log n)$.
▸ Hint 8
Integer ternary search loop: maintain integer bounds lo and hi. Each round compute t1 = lo + (hi−lo)//3 and t2 = hi − (hi−lo)//3; if f(t1) ≤ f(t2) set hi = t2, else set lo = t1. Stop when hi − lo ≤ 2, then brute-check the remaining ≤ 3 candidates. Initial range: lo = 1, hi = max(b) — going above max(b) makes cost_b zero but only inflates cost_a.
▸ Hint 9
Edge case first: after sorting, if a[0] ≥ b[−1] (i.e., min(a) ≥ max(b)), print 0 and return — the condition is already satisfied. Also: element values reach $10^9$ and arrays reach $10^5$, so sums can hit $10^{14}$ — Python handles this natively. Use sys.stdin.buffer.read() for speed on large inputs.
▸ Hint 10
Full Python sketch:
import sys\nfrom bisect import bisect_left, bisect_right\n\ndef solve():\n    data = sys.stdin.buffer.read().split()\n    n, m = int(data[0]), int(data[1])\n    a = sorted(int(x) for x in data[2:2+n])\n    b = sorted(int(x) for x in data[2+n:2+n+m])\n    if a[0] >= b[-1]:\n        print(0); return\n    pa = [0]*(n+1)\n    for i in range(n): pa[i+1] = pa[i] + a[i]\n    pb = [0]*(m+1)\n    for i in range(m): pb[i+1] = pb[i] + b[i]\n    def cost(x):\n        i = bisect_left(a, x)         # a[0..i-1] are all < x\n        ca = i*x - pa[i]\n        j = bisect_right(b, x)        # b[j..m-1] are all > x\n        cb = (pb[m]-pb[j]) - (m-j)*x\n        return ca + cb\n    lo, hi = 1, b[-1]\n    while hi - lo > 2:\n        t1 = lo + (hi-lo)//3\n        t2 = hi - (hi-lo)//3\n        if cost(t1) <= cost(t2): hi = t2\n        else: lo = t1\n    print(min(cost(x) for x in range(lo, hi+1)))\n\nsolve()
bisect_left gives elements strictly less than x (correct for cost_a); bisect_right gives the first index strictly greater than x (correct for cost_b). The final range(lo, hi+1) checks at most 3 values.
▸ Similar 2 codeforces 1700 Increasing Frequency ↗
Tags: dp, prefix sums, strings
Why this problem

Applies the 'fix a target value, replace every element with +1 or −1, then find max subarray sum' transform—precisely the reduction 600/D uses after fixing the median: each a[i] becomes a[i]−median and a prefix-sum max-scan finds the optimal window.

▸ Style (canonical pattern + twist)
Canonical pattern: Maximize an element-frequency count after a uniform-shift-on-segment operation — reduces to maximum subarray sum on a derived ±1 array.
Twist: The shift $k$ is completely free (any integer), which means you are really choosing which value $v = c - k$ to "promote" to $c$ inside the chosen segment. Efficiently iterating over all possible $v$ without blowing up to $O(n^2)$ is the crux.
Builds: Reformulating counting differences as prefix-sum differences; running Kadane's state sparsely — only at positions where the value changes — so total work stays $O(n)$ even with many distinct values.

Hint ladder (12 hints)

▸ Hint 1
Take a tiny example — say $a=[5,2,5,5]$, $c=2$. Trace what happens when you pick segment $[1,4]$ and $k=-3$: exactly which original values become $2$? Which elements outside the segment still contribute to the final count? Notice that all elements inside shift by the same $k$ — what is the relationship between $k$ and which original value converts to $c$?
▸ Hint 2
From hint 1: every $a[i]$ inside $[l,r]$ becomes $c$ exactly when $a[i] = c - k$. That fixes one target value $v = c - k$ per operation; since $k$ is free, $v$ can be anything. Now write out the total count of $c$ after the operation in terms of $v$, the segment, and the original array — how many $c$s come from outside the segment, and how many from inside?
▸ Hint 3
From hint 2: total $c$-count after $=$ [c's outside] $+$ [v's inside] $=$ $(\text{total\_c} - \text{count}(c \text{ in } [l,r])) + \text{count}(v \text{ in } [l,r])$. Rearranging: answer $= \text{total\_c} + \max_{v,\,[l,r]}[\text{count}(v \text{ in } [l,r]) - \text{count}(c \text{ in } [l,r])]$. The whole problem has collapsed into finding the value $v$ and segment that maximize this net gain.
▸ Hint 4
From hint 3: for a fixed $v$, define $b[i] = +1$ if $a[i]=v$, $-1$ if $a[i]=c$, $0$ otherwise. The gain for segment $[l,r]$ is exactly the subarray sum of $b$ over $[l,r]$. You need the maximum subarray sum of $b$. You know an $O(n)$ algorithm for this — what is it, and why can't you just run it independently for each value $v$?
▸ Hint 5
From hint 4: that's Kadane's algorithm, $O(n)$ per value. But there can be $O(n)$ distinct values, making the naive approach $O(n^2)$. Key observation: for a fixed $v$, only positions where $a[i]=v$ (contribute $+1$) and $a[i]=c$ (contribute $-1$) are active in $b$ — all other positions are zero and the running sum is flat there. How can you exploit this to avoid scanning all $n$ elements for each $v$?
▸ Hint 6
From hint 5: precompute $\text{prefix\_c}[i] = \#\{j < i : a[j] = c\}$ once in $O(n)$. For value $v$, the Kadane prefix at index $i$ is $h_v(i) = \text{prefix\_v}(i) - \text{prefix\_c}(i)$. The maximum subarray sum of $b$ equals $\max_{l \le r}[h_v(r) - h_v(l)]$ — the classic best buy/sell profit on a sequence. Now ask: at which indices can $h_v(i)$ actually increase?
▸ Hint 7
From hint 6: $h_v(i)$ increases by 1 only when $a[i-1] = v$, and decreases by 1 only when $a[i-1] = c$. Between two consecutive $v$-positions, $h_v$ is non-increasing (only $c$s can change it, pulling it down). Therefore the minimum of $h_v$ over any inter-$v$ gap is found at the right end of that gap — right before the next $v$ arrives.
▸ Hint 8
From hint 7: let the occurrences of $v$ be at 0-indexed positions $p_1 < p_2 < \cdots < p_k$. At the $j$-th occurrence: $h_v(p_j) = (j{-}1) - \text{prefix\_c}[p_j]$ (just before processing it) and $h_v(p_j{+}1) = j - \text{prefix\_c}[p_j]$ (just after, since $a[p_j]=v\ne c$). The minimum of $h_v$ over all indices $0 \ldots p_j{+}1$ equals $\min\bigl(0,\, h_v(p_1),\, \ldots,\, h_v(p_j)\bigr)$ — updatable with a single running minimum variable.
▸ Hint 9
From hint 8: the best gain for value $v$ is $\max_j \bigl[h_v(p_j{+}1) - \text{running\_min up to }\,p_j\bigr]$. Processing the $k$ positions of $v$ costs $O(k)$. Since every element belongs to exactly one value group, $\sum_v \text{count}(v) = n$, so the total work across all values is $O(n)$ — not $O(n \times |\text{distinct values}|)$.
▸ Hint 10
Full algorithm: (1) Build $\text{prefix\_c}$ in $O(n)$. (2) Group each non-$c$ element's index by its value. (3) For each group, scan positions $p_1, p_2, \ldots$ in order: maintain $\text{running\_min}$ starting at $0$; at the $j$-th position $p$, update $\text{running\_min} = \min(\text{running\_min},\, (j{-}1) - \text{prefix\_c}[p])$, then record candidate gain $(j - \text{prefix\_c}[p]) - \text{running\_min}$. (4) Answer $= \text{total\_c} + \text{best\_gain}$.
▸ Hint 11
Edge cases: skip $v = c$ entirely (gain is always 0). If $\text{best\_gain}$ stays $0$, the answer is just $\text{total\_c}$ — the best move is a no-op. For $n=1$: whether $a[0]=c$ or not, you can always achieve count $1$.
▸ Hint 12
Code sketch (Python):
import sys
from collections import defaultdict

def solve():
    data = sys.stdin.buffer.read().split()
    n, c = int(data[0]), int(data[1])
    a = [int(x) for x in data[2:n+2]]

    prefix_c = [0] * (n + 1)
    for i in range(n):
        prefix_c[i+1] = prefix_c[i] + (1 if a[i] == c else 0)
    total_c = prefix_c[n]

    groups = defaultdict(list)
    for i, x in enumerate(a):
        if x != c:
            groups[x].append(i)

    best_gain = 0
    for pos_list in groups.values():
        running_min = 0
        for j, p in enumerate(pos_list, 1):
            h_before = (j - 1) - prefix_c[p]
            running_min = min(running_min, h_before)
            best_gain = max(best_gain, j - prefix_c[p] - running_min)

    print(total_c + best_gain)

solve()
Use sys.stdin.buffer.read() for speed with $n = 5 \times 10^5$. enumerate(pos_list, 1) gives 1-indexed $j$. Note j - prefix_c[p] is $h_{\text{after}}$ because $a[p]=v\ne c$ so $\text{prefix\_c}[p{+}1]=\text{prefix\_c}[p]$.
